# 旧项目迁移指南

旧项目不要一上来强制套完整流程。推荐先把工作流作为“旁路治理层”接入，跑通后再逐步收敛到标准 SDLC。

## 迁移思路

```text
安装工作流
-> 自动生成 INDEX.md 草稿
-> 用户/项目负责人确认上下文
-> 选择一个小功能或 bugfix 试点
-> 复盘流程负担和产物质量
-> 再决定是否扩大到团队默认流程
```

## 1. 安装到旧项目

中文版本：

```bash
bash scripts/install.sh bootstrap-project /path/to/legacy-project --lang zh
```

英文版本：

```bash
bash scripts/install.sh bootstrap-project /path/to/legacy-project --lang en
```

脚本会：

- 合并或创建 `AGENTS.md`。
- 安装 `specs/` 模板。
- 自动扫描项目并生成 `specs/global/INDEX.md` 草稿。
- 在 INDEX 中标注需要用户确认的字段。

## 2. 确认项目上下文

重点确认：

- 产品名称和业务边界。
- 当前版本、里程碑和进度文档。
- 测试、构建、本地启动命令是否可直接执行。
- 主要源码入口和测试位置是否准确。
- 是否已有 docs/specs 可以接入。

## 3. 第一个试点任务

优先选择：

- 一个低风险小功能。
- 一个可复现 bug。
- 一个技术 spike。

暂时避免：

- 跨多系统重构。
- 高压线上事故。
- 需求仍然高度不确定的大功能。

## 4. 切换原则

- 旧项目已有规范优先保留，`AGENTS.md` 只追加公司工作流段落。
- 旧项目已有 `specs/` 时默认不覆盖。
- 自动生成的 INDEX 是草稿，不是事实来源。
- 第一次试点只要求流程可追溯，不要求模板一次填满。
- 空字段如果连续多次无用，应从项目模板中删减。

## 5. 何时可以正式切换

满足以下条件后，可以把工作流设为团队默认：

- 新功能能从需求追溯到验证。
- bugfix 能记录根因和回归验证。
- 团队接受新增文档成本。
- Codex 能从 INDEX 进入项目状态，减少重复解释。
- 专家 bundle 没有频繁误触发。
