# Pilot Playbook

这份手册用于公司内部试点。目标是在一个真实项目里验证工作流是否提升交付稳定性，而不是一次性强推到所有团队。

## 试点范围

建议选择：

- 一个中等复杂度 feature。
- 一个真实 bugfix。
- 一个可选 spike 或 hotfix 演练。

暂不建议选择：

- 跨多个系统的大型重构。
- 尚未明确业务目标的长期项目。
- 高风险线上事故的第一次真实处置。

## 参与角色

- 业务/产品确认人：确认需求和验收标准。
- 技术负责人：确认设计、边界和风险。
- 实现人：使用 Codex 执行任务。
- Reviewer：检查输出、验证证据和流程负担。

## 第一个 Feature 怎么跑

1. 业务用户说：`帮我梳理这个功能需求：...`
2. 需求确认后说：`需求已确认，进入技术设计`
3. 技术负责人确认设计后说：`方案已确认，进入任务拆解`
4. 实现人确认任务后说：`任务已确认，开始实现`
5. 完成后要求 Codex 输出：
   - 变更总结
   - 文件变更
   - 验证命令或手工验证
   - 剩余风险

## 第一个 Bugfix 怎么跑

1. 用户说：`开始 bugfix：...`
2. Codex 先判断 bug 还是需求变更。
3. Codex 复现或收集证据。
4. Codex 做最小修复。
5. Codex 输出根因、修复、回归验证。

## 试点验收指标

- 需求是否比以前更清楚。
- 设计和实现是否能追溯。
- 验证证据是否更完整。
- 简单任务是否没有被过度流程化。
- 专家技能是否在合适节点自动参与。
- 团队是否能接受新增文档成本。

## 试点结束复盘

建议复盘以下问题：

- 哪些唤起语最常用？
- 哪些节点太重？
- 哪些专家 bundle 缺失或误触发？
- 哪些模板字段没人填？
- 是否可以进入第二个项目试点？
