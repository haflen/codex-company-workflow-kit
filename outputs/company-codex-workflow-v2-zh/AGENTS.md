# 公司 Codex 工作流

本项目使用轻量 SDLC。目标是让需求、设计、实现和验证可追溯，同时不拖慢简单工作。

## 上下文优先

- 规划或编辑前，先阅读 `specs/global/INDEX.md`。
- 如果存在 `说明文档.md` 或等价进度文档，读取当前进展和项目状态。
- 只读取当前版本、里程碑、功能文档和相关源码。
- 除非任务是审计或迁移，不要扫描所有 Markdown 文件。

## 工作流文档职责

写入任何项目文档前，必须先确认文档角色、信息层级、编号命名空间和更新触发条件。不要把项目入口摘要、spike 现场流水、正式需求/设计/任务和生命周期总结混成同一条任务链。

默认职责：

- `说明文档.md` 或等价入口页：项目入口、当前状态、最近重要事件、阅读路线。不要承载 spike 现场流水，不使用 spike 内部任务号。
- `spike_*_工作日志.md` 或 spike 目录日志：spike 现场流水、实验过程、临时结论。使用 spike 内部编号，例如 `SPK02-T001`，不要使用裸 `任务 001`。
- `specs/versions/...` 或 `specs/features/...`：正式需求、设计、任务计划和验收依据。使用 feature、版本或正式任务编号，不继承 spike 流水号。
- `docs/lifecycle/`：版本生命周期阶段总结和里程碑回顾，只写阶段性摘要，不追逐每个小任务。
- `specs/global/INDEX.md`：文档职责地图和路由索引，是判断该写哪个文档的入口。

更新触发：

- 项目入口页只在当前阶段、最近重要事件、阅读路线或关键状态变化时更新。
- Spike 日志只在 spike 实验、观察、决策或临时任务变化时更新。
- 正式 specs 只在需求、设计、任务、验收或正式变更被确认时更新。
- 生命周期文档只在版本阶段结束、里程碑变化或需要管理层摘要时更新。

发现同一裸编号出现在不同层级文档时，必须先暂停写入并建议改成带命名空间编号，例如 `SPK02-T183`、`FEAT-DT-T01` 或日期型项目事件。

## 多分支公共文档协议

多分支并行时，公共入口文档默认代表主线事实，不代表某个开发分支的局部状态。

- `说明文档.md`、`specs/global/INDEX.md` 等公共主线文档默认只在 `main`、`master`、`develop`、`integration` 或明确的集成分支更新。
- feature、spike、hotfix 分支默认不直接把未合并内容写入公共文档的“当前状态”。
- 分支如果影响项目入口、当前阶段、最近重要事件、阅读路线、文档职责或编号规则，应写入 `docs/public-doc-updates/<branch-or-feature>.md`。
- 分支私有过程写入 `specs/features/<feature>/`、`specs/versions/...`、`spikes/` 或对应工作日志，不写进公共入口页。
- 合并阶段统一读取公共文档影响补丁，再把已合并事实升格到 `说明文档.md` 和 `specs/global/INDEX.md`。
- 多个分支修改同一公共段落时，不在业务分支抢改公共文档；先保留各自补丁，由集成分支统一重写公共段落。
- 如确需在业务分支直接改公共文档，必须说明原因，并在完成报告中标记为 `公共文档直接写入风险`。

## 阶段边界

- 需求阶段产出目标、范围、验收标准和边界条件，不编辑实现代码。
- 设计阶段产出架构、契约、数据流、风险和测试策略，不编辑实现代码。
- 规划阶段产出可执行任务和验证点，不编辑实现代码。
- 实现阶段只在需求、设计和任务确认后开始，`/spike` 和 `/hotfix` 除外。
- 模糊的“继续”表示继续当前阶段，不表示进入下一阶段。

## 第一性原理与对抗式审查

公司 workflow 内置两类检查，不要求用户每次手动输入提示词：

- `First Principles Check`：回到底层事实、约束和最小成立条件，避免用经验类比替代表层问题的根因分析。
- `Adversarial Review`：从恶意用户、极端数据、异常状态、权限绕过、并发重试、超大输入、未来时间、缓存假阳性和渲染压力等反例角度验证方案。

自动触发规则：

- 需求阶段：L2/L3 或业务规则复杂时，至少写出核心假设、不可破坏约束和反例场景。
- 设计阶段：架构、数据、权限、性能、安全、外部 API、前端渲染或跨服务边界不平凡时，必须执行 `First Principles Check`。
- 规划阶段：每个非平凡任务必须包含最小失败案例或验证锚点，并至少包含一个对抗场景。
- 实现阶段：完成前必须执行 `Adversarial Review`，除非纯文案、注释或无行为小改动，并说明原因。
- Bugfix/hotfix：定位根因前必须执行 `First Principles Check`，完成前必须执行回归和对抗式审查。
- Spike：实验结论前必须写出第一性原理假设、反例实验和证据边界。
- 技能升级、安全审查、自进化提案默认使用 `full-audit`，并执行对抗式审查。

输出要求：

- `第一性原理检查：` 写底层事实、约束、最小成立条件或为什么本轮可跳过。
- `对抗式审查：` 写极端、恶意或异常场景、结果或为什么本轮可跳过。
- 不要只写“已从第一性原理分析”或“已做对抗式审查”；必须给出具体事实或场景。

## Superpowers 可见叠加

- 每个 workflow 输出都必须显式包含 `工作流层`、`Superpowers 叠加` 和 `执行策略`。
- 每个 workflow 开头都必须显式包含 `实际调用`、`专家/插件能力` 和 `未调用但采用视角`，不能只写“专家视角”而不说明是否真实调用 skill。
- 每个 workflow 结尾都必须显式包含 `验证证据`、`未验证项` 和 `剩余风险`。如果本阶段不需要验证，必须说明原因。
- 如果某个 Superpowers、专家 skill、MCP、浏览器或其他 Codex 插件能力没有在当前会话真实暴露或真实调用，必须写入 `未调用但采用视角`，并说明原因。
- 需求探索默认叠加 `superpowers:brainstorming`。
- 复杂任务拆解默认叠加 `superpowers:writing-plans`。
- 实现阶段默认叠加 `superpowers:test-driven-development` 和 `superpowers:verification-before-completion`。
- Bugfix 默认叠加 `superpowers:systematic-debugging`，完成前叠加 `superpowers:verification-before-completion`。
- 如果某个简单任务不叠加 Superpowers，必须明确说明原因。

## 能力调用透明度

所有公司 workflow 默认使用以下透明度协议，除非用户明确要求极简输出：

### 自动分级

用户不需要判断使用哪种透明度级别；workflow 必须自动判定：

- `light`：默认模式。用于普通阶段内推进、小改动、文档更新、低风险继续工作。只保留必要的实际调用、仅采用视角、验证证据和风险说明。
- `full-audit`：关键节点或异常模式。命中以下任一条件时自动启用完整 `Workflow Audit`：
  - 阶段交接：需求到设计、设计到任务、任务到实现。
  - 实现完成、bugfix 完成、hotfix 任意阶段、spike 结论输出。
  - 技能升级、专家依赖更新、安全审查、自进化提案。
  - 出现“专家/插件未真实调用，只采用视角”。
  - 当前会话缺少应有 Superpowers、专家 skill、MCP、浏览器或插件能力。
  - 验证失败、验证缺失、无法运行测试。
  - 涉及生产、数据、权限、架构、性能或安全风险。
  - 用户要求“审计”“检查流程”“确认是否合规”。

开头：

- `工作流层：`
- `透明度模式：light` 或 `透明度模式：full-audit`
- `实际调用：` 写真实触发或读取的 workflow、Superpowers、专家 skill、MCP、浏览器能力。
- `专家/插件能力：` 写本轮需要或已选择的专家、Superpowers、Codex 插件能力。
- `未调用但采用视角：` 写当前会话不可见、阶段不适合或风险不值得真实调用的能力。
- `第一性原理检查：`
- `对抗式审查：`
- `执行策略：`

结尾：

- `验证证据：` 写命令、检查结果、文件变更、截图、日志或人工检查证据。
- `未验证项：` 写未能验证或本阶段无需验证的内容。
- `剩余风险：`

不能用“已按专家视角执行”替代真实调用说明；必须区分 `已真实调用` 与 `仅采用视角`。

`full-audit` 模式额外输出：

- `Workflow Audit：`
- `阶段边界：`
- `Superpowers 声明：`
- `专家/插件真实调用：`
- `仅采用视角：`
- `验证证据：`
- `未验证项：`
- `结论：`

## 专家使用

普通工作不要求用户手动点名专家。工作流应从 `BUNDLES.md` 选择最小匹配组合，再在当前阶段需要时使用对应专家。

如果用户明确点名专家，在不违反安全和项目约束的前提下优先遵循。

第三方 API 或框架行为可能变化时，优先使用官方最新文档。没有 Context7 类 MCP 时，说明限制并使用官方文档或本地包文档。

使用 `company-expert-routing` 作为专家选择入口。使用 `BUNDLES.md` 管理专家组合，使用 `EXPERTS.lock.md` 检查外部专家技能的来源、pin、许可和审查状态。优先读取项目根目录文件；如果项目内缺失，读取插件内置 fallback，不要直接放弃专家路由。

使用模板时优先读取项目内 `specs/global/assets/`；如果项目模板缺失，读取插件内置 fallback，并提醒用户执行 `bootstrap-project` 或 `update-templates` 补齐项目模板。

不要自动更新外部专家技能。采用 `company-skill-upgrade-runner` 获取候选版本、比对新旧内容、安全审查、输出建议，并等待用户明确确认后再覆盖。

自进化默认只产出提案。使用 `company-skill-evolution-lab` 记录重复问题并提出受控改进，不允许 skill 直接自改生产版本。

## 验证

- 功能工作在有测试框架时需要测试。
- Bug 修复在改代码前需要复现或清晰证据链。
- 完成报告必须包含准确验证命令或手工检查。
- 无法自动验证时，提供聚焦的手工检查清单。

## 文档

- 公司项目应维护 `说明文档.md` 或等价进度文档。
- 项目应维护文档职责地图，优先写在 `specs/global/INDEX.md`。
- 不同层级文档不得共享裸任务编号；spike、feature、版本任务和项目事件必须使用不同命名空间。
- 版本化 specs 可使用 `specs/versions/<version>/<milestone>/`。
- 小变更可使用 `specs/features/<feature>/` 下的紧凑 spec。
- 公共 API 和复杂逻辑需要注释；常规函数不需要样板注释。
