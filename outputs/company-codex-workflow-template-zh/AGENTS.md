# Codex 公司项目工作流

这些说明定义本项目中的 Codex 团队工作流。所有变更应保持简单、聚焦、可验证。

## 工作原则

- 优先选择满足已确认需求的最小改动。
- 引入新模式前，先匹配现有项目风格和架构。
- 交付功能或修复问题时，不顺手重构无关代码。
- 当需求、归属或风险不清楚时，明确说明假设。
- 对变化较快的第三方 API，优先使用当前官方文档，不依赖记忆。

## 上下文加载

- 先阅读 `specs/global/INDEX.md`。
- 只阅读与当前任务相关的具体 spec。
- 除非任务明确要求审计或迁移，不要扫描所有 Markdown 文件。
- 如果索引过期，在规划阶段或完成相关 spec 变更后更新索引。

## 阶段边界

- 需求澄清阶段产出验收标准和边界条件，不编辑实现代码。
- 技术设计阶段产出架构、API/数据契约、风险说明和测试策略，不编辑实现代码。
- 实现阶段只在需求和设计确认后开始，除非用户明确使用 `/spike` 或 `/hotfix`。
- 模糊的“继续”表示继续当前阶段，不表示直接进入编码。

## 标准交接口令

- `需求已确认，进入技术设计`：从需求进入技术设计。
- `方案已确认，开始实现`：从设计进入实现。
- `开始 bugfix`：进入根因分析和修复。
- `/spike <问题>`：进入限时技术探索。
- `/hotfix <事故>`：进入紧急修复。

## 验证规则

- 项目有测试框架时，功能交付应包含可执行测试。
- Bug 修复应先复现或解释失败原因，再改代码。
- 完成声明必须引用实际执行的验证命令或手工检查。
- 如果无法自动验证，提供聚焦的手工检查清单。

## 例外通道

- `/spike` 可以跳过完整 SDLC 文档，但必须产出简短 spike 报告。
- `/hotfix` 可以优先恢复服务，但必须产出 hotfix 报告和后续测试债务。
- 例外工作应最小化、隔离，并在事后进入正常清理规划。
