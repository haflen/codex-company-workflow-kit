# Company Skill Maintenance

## Purpose

Treat expert skills like dependencies: pinned, reviewed, updated deliberately, and reversible.

## Workflow

1. For ordinary user-initiated updates, route to `company-skill-upgrade-runner`.
2. Read `EXPERTS.lock.md`.
3. Identify the skill to add, update, replace, or remove.
4. Check `BUNDLES.md` to understand which workflows depend on the skill.
5. Inspect source, license, pin, changed files, and risky instructions.
6. Run `company-skill-security-review` before adopting external changes.
7. Compare old and new trigger descriptions for drift; local Codex `description: Use when...` frontmatter is protected by default.
8. Confirm the sync scope covers only referenced or explicitly named external expert skills, not the entire upstream repository.
9. Update `EXPERTS.lock.md` only after review.
10. Update `BUNDLES.md` only if bundle membership or default routing changes.
11. Record release notes and rollback instructions.

## Update Policy

- Company default: monthly review or immediate review after major framework/runtime upgrades.
- Security or supply-chain concern: review immediately.
- Do not auto-merge skill updates from external hubs.
- Do not introduce new external experts without a lock entry and review.
- External expert updates preserve local trigger frontmatter, company routing rules, and Superpowers layering by default.
- For large upstream repositories, sync only the target list; prefer raw/API/sparse to avoid full-download time and bandwidth waste.
- Reports generated by install or verification are artifacts; do not commit them unless they are intentionally designed as templates.
- Script executable bits, sandbox risk, and runtime approval requirements are part of dependency state and must be maintained.
